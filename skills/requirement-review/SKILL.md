# 需求评审技能

## 你的任务
阅读需求文档，从测试视角评审其质量，输出结构化 JSON。

## 评审维度

### 1. 完整性
- 功能边界是否清晰（入口/出口/边界值是否有定义）
- 异常流是否覆盖（错误码、失败场景）
- 前置条件和依赖是否说明

### 2. 一致性
- 同一术语是否前后统一
- 是否存在相互矛盾的描述

### 3. 可测性
- 每条需求能否写出可验证的预期结果
- 标记出"无法测试"的模糊需求（如"系统应该快速响应"）

### 4. 风险识别
- 标记安全相关需求（鉴权、数据隔离）
- 标记性能相关需求（并发、超时）
- 标记与其他系统的集成点

## 输出格式（严格按此 JSON 输出，不要输出其他内容）

```json
{
  "score": 85,
  "summary": "一句话描述需求整体质量",
  "completeness_issues": [
    "缺少 XXX 的异常处理描述"
  ],
  "consistency_issues": [
    "第2节称'用户'，第5节称'账户'，指同一概念"
  ],
  "untestable_items": [
    "「系统响应应迅速」—— 缺少具体指标"
  ],
  "risk_flags": [
    { "type": "security", "desc": "登录接口未说明防暴力破解机制" },
    { "type": "performance", "desc": "未定义并发用户数上限" },
    { "type": "integration", "desc": "依赖第三方支付，未说明超时处理" }
  ],
  "testable_features": [
    "用户注册",
    "登录/登出",
    "密码重置"
  ]
}
```

## 注意
- score 范围 0-100，低于 60 说明需求质量较差，建议退回完善
- issues 列表为空时输出空数组 []，不要省略字段
- testable_features 是后续生成测试点的输入，尽量细化到功能点级别
